## 2. Critical Gaps (What's Missing)

### 2.1 Approval Routing Engine — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Approval Routes):
> Multi-step approval configuration by amount, department, and category. CSV import supported.

**Current State**: No routing logic exists. A manager directly approves/rejects — no step sequencing, no route matching by amount threshold or department rules.

**What's Needed**:
- New tables: `approval_routes`, `approval_route_steps`
- Route matching logic: given (amount, department, category) → determine step sequence
- Admin UI for route configuration
- CSV import for bulk route setup
- Emergency ringi route (post-hoc approval)

**Impact**: This is the **core differentiator** of Phase 2. Without it, the system cannot scale beyond simple manager approval.

---

### 2.2 Multi-Step Approval Workflow — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Approval Operations):
> Statuses: Draft / Submitted / Pending Approval (per step) / Sent Back / Rejected / Approved / Expired

**Current State**: Flat status model. `approved_by` and `rejected_by` are single UUID fields — no concept of "which step approved."

**What's Needed**:
- New table: `approval_step_history` (step_order, approver_id, action, comment, timestamp)
- Columns on `approval_requests`: `current_step_index`, `approval_route_id`, `next_approver_ids`
- Step-by-step state machine with transition validation
- Per-step approver assignment and tracking

---

### 2.3 Notification System — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Notifications & Chat Integration):
> Email and Slack DM notifications. Notification center with pending count. Approve/send-back from chat card.

**Current State**: Navbar notification bell shows hardcoded `"2"` (`src/components/layout/Navbar.tsx`). No actual notification delivery exists.

**What's Needed**:
- Email notification service (approval requests, reminders, completions)
- Slack integration (DM notifications, interactive approval cards)
- Notification center component with real pending count
- Notification preferences per user
- n8n workflow integration for notification delivery

---

### 2.4 Deadline & Escalation Management — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Approval Operations):
> Reminders at 24h and 72h; escalation on overdue.

**Current State**: No `due_at` column. No reminder or escalation logic.

**What's Needed**:
- Column: `due_at` on `approval_requests`
- Column: `escalated_to` for escalation tracking
- Scheduled job (cron or n8n) for reminder checks
- Escalation rules and notification triggers
- UI: due date display on approval cards, overdue indicators

---

### 2.5 Proxy Approval — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Approval Routes):
> Proxy approval limited by period / scope / amount ceiling.

**Current State**: Not modeled at all.

**What's Needed**:
- New table: `proxy_approvals` (delegator, delegate, valid_from, valid_to, max_amount, scope)
- Logic to check proxy authority before allowing approval action
- Audit trail distinguishing proxy vs direct approval

---

### 2.6 Reporting Dashboard — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Reporting):
> Dashboard: approval lead time, send-back rate, count by category, delay rankings. CSV/Excel export.

**Current State**: `activity_logs` has basic stats (count by action type, date range). No ringi-specific metrics.

**What's Needed**:
- Lead time calculation (submitted_at → approved_at per request)
- Send-back rate aggregation
- Category breakdown charts
- Delay ranking (longest pending requests)
- CSV/Excel export with IP, user, operation details

---

### 2.7 PDF Generation — NOT IMPLEMENTED

**Spec Requirement** (Section 5: Business Flow):
> Upon completion, the ringi PDF and attachments are stored in S3 with metadata applied.

**Current State**: Attachments are linked but no PDF is generated on completion.

**What's Needed**:
- PDF generation service (e.g., using puppeteer, react-pdf, or jsPDF)
- Auto-generate ringi summary PDF on final approval
- Store in S3 with metadata tags
- Link to approval request record

---

### 2.8 Attachment Versioning & Tamper Detection — NOT IMPLEMENTED

**Spec Requirement** (Section 6: Submission, Approval Operations):
> Attachments are tamper-proof after approval. Hash stored. Version tracking.

**Current State**: `approval_request_documents` has basic linking. No hash, no version field.

**What's Needed**:
- Columns: `hash` (SHA-256), `version` on document records
- Hash computation on upload, verification on download
- Version history display
- Lock attachments after approval (prevent modification)

---

### 2.9 Chat-Initiated Submission UX — PARTIALLY EXISTS

**Spec Requirement** (Section 9: UI/UX):
> Open the submission form from chat via a modal or drawer. Context-aware input assistance.

**Current State**: OpenAI tool `manage_approval_requests` can create requests from chat. But there's no dedicated modal/drawer triggered from chat UI. `PendingApprovalRequests` component shows pending items in sidebar.

**What's Needed**:
- Drawer/modal component triggered from chat
- Context-aware pre-population (summarize chat messages → prefill form)
- Chat card with Approve/Send Back/Comment action buttons

## 3. Spec vs Code Mismatches

### 3.1 Status Model Mismatch

| Spec Status | Code Status | Match? |
|-------------|-------------|--------|
| Draft | *(not implemented)* | MISSING |
| Submitted | `pending` | Partial — spec separates "submitted" from "pending approval" |
| Pending Approval (per step) | `pending` | MISMATCH — code has no step concept |
| Sent Back | `need_revision` | MISMATCH — see 3.2 |
| Rejected | `rejected` | OK |
| Approved | `approved` | OK |
| Expired | *(not implemented)* | MISSING |
| Cancelled | `cancelled` | OK (extra, not in spec but useful) |

**Action Required**: Refactor status enum. Add `draft`, `submitted`, `sent_back`, `expired`. Rename `need_revision` → `sent_back` for spec alignment.

---

### 3.2 Send-Back vs Rejection Conflation

**Spec**: Send-back and rejection are distinct operations:
- **Send-back**: Returns to requester for correction, comments required, can be re-submitted
- **Rejection**: Terminal state, comments required, request is closed

**Code**: `need_revision` is used ambiguously. The UI treats it similarly to rejection. There's no re-submission flow with diff memo.

**Action Required**: Separate workflows. Send-back should reset to requester with editable state. Rejection should be terminal.

---

### 3.3 Approver Tracking Mismatch

**Spec**: Step-based tracking with `current_approver_ids`, `next_approver_ids`, step order.

**Code**: Flat fields `approved_by` (single UUID) and `rejected_by` (single UUID).

**Action Required**: Replace flat fields with relational step tracking. Each step records who approved/rejected and when.

---

### 3.4 Audit Log Field Gap

**Spec** (Section 6: Permissions & Security):
> Record operation type, before/after diff, IP, user agent, and timestamp — stored in tamper-proof area.

**Code** (`activity_logs` table):
- Has: `action`, `old_values`, `new_values`, `metadata` (JSONB)
- Missing: `ip` column, `user_agent` column
- No tamper-proof enforcement (no write-once constraint)

**Action Required**: Add `ip` and `user_agent` columns. Consider append-only table policy (deny UPDATE/DELETE via RLS).

---

### 3.5 Form Validation Library Inconsistency

**Auth form** (`src/components/auth/AuthForm.tsx`): Uses React Hook Form + Zod
**All other forms** (approval requests, user management): Use Formik + Yup

This isn't a spec mismatch but a codebase inconsistency worth noting. The AGENTS.md convention says "Formik + Yup validation" — auth form deviates.

## 4. Architectural Risks

### 4.1 n8n Dependency for Workflow
Spec says "use existing n8n" for workflow engine. Current codebase has n8n integration for RAG. Need to clarify:
- Will approval routing logic live in n8n or in app code?
- How will n8n trigger step transitions, reminders, and escalations?
- What happens if n8n is down? (Queue persistence requirement)

### 4.2 No Queue/Job System
Spec requires queue persistence, backoff retry, and idempotency keys. Current app has no background job infrastructure beyond n8n webhooks.

### 4.3 Tenant Isolation Not Verified
Spec requires workspace/tenant isolation. Current RLS policies scope to `user_id` but multi-tenant (workspace-level) isolation hasn't been verified for approval routes shared across a team.

### 4.4 Data Retention
Spec requires 7-year retention with export API. No retention policy or archival mechanism exists.

## 5. Strengths to Build On

1. **Solid server action pattern** — auth check → role check → operation → return `{ data, error }`. Easy to extend for new ringi services.
2. **Activity logging foundation** — `logActivity()` fire-and-forget pattern is already used throughout. Just needs IP/user-agent enrichment.
3. **i18n coverage** — 2100+ translation keys already cover approval-related terms. Phase 2 additions will extend, not replace.
4. **OpenAI tool integration** — `manage_approval_requests` tool already exists. Can be enhanced for step-aware operations.
5. **Component library** — Approval detail page, submission dialog, timeline component, filter panel all exist as starting points.
6. **Type-first approach** — All types in `src/types/`. Clean extension point for new models.

## 6. Recommended Implementation Order

Based on dependency analysis:

1. **Database schema expansion** — Add missing columns and new tables (routes, steps, proxy)
2. **Status model refactor** — Align statuses with spec (add draft, submitted, sent_back, expired)
3. **Approval routing engine** — Route configuration + matching logic
4. **Multi-step workflow** — Step-based state machine + per-step tracking
5. **Send-back workflow** — Distinct from rejection, with re-submission + diff memo
6. **Deadline & escalation** — due_at, reminders, escalation triggers
7. **Notification system** — Email + Slack via n8n
8. **Chat integration** — Modal/drawer from chat, approval action cards
9. **Reporting dashboard** — Lead time, send-back rate, category breakdown
10. **PDF generation** — Completion PDF with S3 storage
11. **Attachment hardening** — Hash, versioning, tamper detection
12. **Proxy approval** — Delegation configuration + authority checks
13. **Audit enhancements** — IP, user-agent, tamper-proof storage

## 7. Open Questions (Need Clarification)

1. Should the approval routing engine be built in **app code** (server actions + DB) or delegated to **n8n workflows**?
2. What are the specific **amount thresholds** for route branching? (Spec Section 13: Open Item)
3. Is `cancelled` status (exists in code, not in spec) intentional and should it be kept?
4. Should the `need_revision` → `sent_back` rename be a breaking migration or additive?
5. What email provider will be used for notifications? (SendGrid, SES, etc.)
6. Is Slack integration via Slack API directly or through n8n?
7. How should "tamper-proof" audit storage be implemented? (Append-only table? Separate DB? Blockchain hash chain?)
