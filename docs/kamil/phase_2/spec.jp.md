# 稟議承認フロー 要件定義書 v0.2

## 1. 目的・背景
- 現行ファイル管理AIエージェントに稟議承認フローを組み込み、チャット起点で稟議を完結させる。
- ベンチマークは freee の稟議承認フローとし、同等の統制・証跡レベルを目標にする（ただし外部連携は行わない）。
- 稟議起票漏れ・承認遅延・履歴散在を解消し、監査対応を簡素化する。

## 2. 適用範囲（スコープ）
- 対象: 購買・契約・経費等の稟議起票、承認、差戻し、却下、代理承認、完了・失効管理。
- チャネル: 本システムWebチャットUI（既存）、メール/Slack通知。
- 除外: freee とのシステム連携（ベンチマークのみ）、新規ワークフローエンジン開発（既存 n8n で対応）、電子署名。

## 3. 利害関係者・ロール
- 起票者: 稟議作成・提出、ドラフト修正、ステータス確認。
- 承認者: 担当稟議の承認/差戻し/却下、コメント入力。代理承認権限あり。
- 管理者: 承認ルート設定、権限管理、監査ログ出力、通知運用。
- 経理/内部統制: リードタイム/差戻し率レポート確認、監査対応。

## 4. 前提・制約
- 認証: OIDC/SAML による SSO。ロール（管理者/承認者/起票者/閲覧者）は本システムで管理。
- ベンチマーク: freee のフロー・ガバナンスを参照し、同等のUX/統制を目指すがシステム連携はしない。
- テナント分離: 既存のワークスペース/テナント分離ポリシーを踏襲。

## 5. 業務フロー概要
1. 起票者がチャットから稟議起票フォームを開き、必須項目と添付を入力して申請。
2. 金額/部門/カテゴリで承認ルートを決定（設定ファイル/管理UIで管理）。承認依頼をメール/Slack通知。
3. 承認者がチャットカード/承認画面で承認/差戻し/却下。差戻し/却下時はコメント必須。
4. ステータスは本システム内で一元管理。完了後に稟議PDFと添付を S3 に保管し、メタデータを付与。
5. 操作ログを保存し、検索・レポート・エクスポートに利用。

## 6. 機能要件
- 起票
  - フォーム項目: 件名、カテゴリ（購買/契約/経費/雑費等）、金額（税込/税抜）、税区分、支払条件、ベンダー/相手先、目的、期間、予算コード、プロジェクト、希望決裁日、申請者/部署。
  - 添付: 見積/契約/関連資料。承認後は改ざん不可（新バージョンで再申請）。
  - 文脈入力支援: チャットメッセージを要約してフォーム初期値を提示。
- 承認ルート
  - 金額・部門・カテゴリ別の承認段階設定（設定ファイル/管理画面で編集・CSV インポート）。
  - 代理承認（期間/対象限定、上限額設定）、緊急稟議ルート（後追い承認必須）。
- 承認操作
  - ステータス: Draft / 申請中 / 承認待ち(段階別) / 差戻し / 却下 / 承認済 / 失効。
  - 差戻し/却下理由のコメント必須。再申請時の差分メモ欄。
  - 期限管理: 24h/72h リマインド、期限超過でエスカレーション。
  - 添付プレビュー、ハッシュ保存による改ざん検知。
- 通知・チャット連携
  - 承認依頼/リマインド/差戻し/完了をメール・Slack DM 配信。チャットカードから承認/差戻し。
  - 通知センターで未処理件数表示。
- 検索・参照
  - 稟議ID、件名、申請者、部門、金額レンジ、ステータス、期間で検索・フィルタ。
  - 承認履歴タイムラインを表示。Drive 上の添付リンクとメタデータを関連付け。
- レポート
  - 承認リードタイム、差戻し率、カテゴリ別件数、遅延ランキングをダッシュボード化。
  - CSV/Excel エクスポート（監査用、IP/ユーザ/操作付き）。
- 権限/セキュリティ
  - RBAC: 起票者は自分の申請のみ編集/閲覧、承認者は担当案件操作、管理者は全件閲覧・設定変更。
  - テナント隔離、最小権限での Drive アクセス。IP 制限オプション。
  - 監査ログ: 操作種別/前後差分/IP/UA/時刻を保存し改ざん防止領域へ。

## 7. 非機能要件
- 可用性: 営業時間帯 99.9%。承認操作応答 3 秒以内、検索 5 秒以内目標。
- 信頼性: 障害時のキュー保持とバックオフリトライ。冪等キーによる二重送信防止。
- セキュリティ: TLS1.2+、SSO、機密度タグに応じた閲覧制御、添付ハッシュ保存。必要に応じ暗号化保管。
- 監査/保持: 監査ログ・稟議データ・添付は原則 7 年保管（要件に応じ設定）。エクスポートAPI/管理UI。
- 運用: メトリクス（リードタイム、差戻し率、エラー率）、アラート閾値（失敗率>2%/5分など）、ランブック整備。

## 8. データ項目（概要）
- 稟議: id, title, category, amount_tax_excl, tax, amount_tax_incl, currency, vendor, purpose, start_date, end_date, payment_terms, budget_code, project_code, requester_id, department, desired_approval_date, created_at, updated_at
- 状態: status, step_order, current_approver_ids, next_approver_ids, due_at, escalated_to, rejection_reason, sendback_reason
- 添付: file_id, s3_object_key, hash, version, created_by, created_at
- 監査: actor_id, action, target_type, target_id, diff_json, ip, user_agent, occurred_at
- 参照用ベンチマーク: freee の稟議ステップID/名称等をマッピング表で保持（外部連携なし、差異レビュー用）。

## 9. UI/UX要件
- チャットからモーダル/ドロワーで起票フォームを開く。必須項目バリデーション、進捗ガード。
- 承認カードに主要項目（件名/金額/相手先/期日）と操作ボタン（承認/差戻し/コメント）。進行度バーと期日表示。
- 添付プレビュー（PDF/画像）とバージョン表示。モバイルでも承認操作可能。
- アクセシビリティ: キーボード操作、色覚配慮配色、エラーメッセージの明確化。

## 10. 運用・管理要件
- 承認ルートを管理画面で編集/CSV インポート。テスト送信でルート確認。
- 通知設定（チャネル/頻度）の管理。メンテナンスモード時は申請受付を停止し告知。
- 定期レポートの自動生成・配信（経理/内部統制向け）。バックアップ日次、RPO<=24h, RTO<=4h 目標。

## 11. 移行要件
- 既存稟議データの取り込みが必要な場合は範囲・形式を定義（ベンチマークとして freee の設計を参照）。ID マッピングと整合性チェックを行う。
- 並行稼働期間を設け、整合性チェックとユーザトレーニングを実施。FAQ/簡易マニュアル配布。

## 12. リスクと対応
- ベンチマークとの差異: freee フローとの差異を定期レビューし、必要な改善を計画。
- 承認遅延: リマインド/エスカレーションルールとダッシュボード監視。
- 権限設定ミス: 週次でロール差分チェックを自動実行。
- 監査要件未確定: 監査チームと必須項目/保管期間/改ざん防止方式を早期合意。

## 13. 未決事項
- 金額閾値と部門別ルートの具体値、緊急稟議の適用条件。
- 代理承認のルール（期間、対象、上限額）。
- ベンチマーク（freee）の具体的な承認ステップ/ガバナンス項目の採用範囲。
- 添付の暗号化・外部共有ポリシー、監査ログの保管先とフォーマット。
