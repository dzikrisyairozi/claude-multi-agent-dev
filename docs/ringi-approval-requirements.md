# 稟議承認フロー導入 要件定義書 v0.2

freee の稟議承認フローをベンチマークとして、現行のファイル管理AIエージェント（以下「本システム」）にチャット起点の稟議作成・承認・証跡管理を実装するための要件整理。freee とのシステム連携は行わない。

## 1. 目的とゴール
- 稟議起票〜承認〜完了までを本システム上で完結し、freee 相当の統制・証跡を実現。
- チャット/ファイル管理体験と稟議をシームレスに接続し、起票漏れや承認遅延を低減。
- 監査・内部統制に耐える履歴・アクセス制御を提供。

## 2. スコープ
- 対象: 稟議（購買/契約/経費等）の起票・承認・差戻し・却下・代理承認・決裁済み管理。
- インターフェース: WebチャットUI（既存）とメール/Slack通知。
- 非スコープ: freee 等との外部連携（ベンチマークのみ）、新規ワークフローエンジン開発（既存 n8n で構成）、電子署名。

## 3. 関連システム・前提
- 本システム: Next.js フロント + API/ワーカー + n8n（ワークフロー/通知） + Google Drive 連携。
- 認証/権限: OIDC（社内ID基盤）で SSO。ロールは「管理者/承認者/起票者/閲覧者」を本システムで管理。

## 4. ペルソナ・ロール
- 起票者: 稟議作成・添付・提出、ステータス確認・修正。
- 承認者: 承認/却下/差戻し、コメント入力。代理承認が可能（委任期間/対象限定）。
- 管理者: ワークフロー定義、承認ルート変更、権限管理、監査証跡のエクスポート。
- 経理/内部統制担当: 承認リードタイム・差戻し率レポート出力、監査対応。

## 5. 業務フロー（概略）
1. 起票者がチャットから `/ringi` などのコマンドまたはUIボタンで稟議起票フォームを開く。
2. 必要項目入力・ファイル添付（契約書/見積書など）。保存時にドラフトとして本システムに保存。
3. ワークフローエンジン（n8n 予定）が所属・金額・カテゴリに応じて承認ルートを決定し、承認依頼を通知（メール/Slack）。
4. 承認者はチャットカードまたは承認画面から承認/却下/差戻し。コメント必須（差戻し/却下時）。
5. ステータスを本システムで一元管理。承認完了後に S3 へ最終稟議PDFと添付を格納し、メタデータを検索可能にする。
6. 監査用に全操作ログ（誰が/いつ/どこから/何を）をイベントとして保存。レポート出力・エクスポート可能。

## 6. 機能要件
- 起票・編集
  - 稟議フォーム（カテゴリ: 購買/契約/経費/雑費 等）。
  - 必須項目: 件名、金額（税込/税抜）、支払条件、ベンダー/相手先、目的、期間、予算コード、関連プロジェクト、希望決裁日。
  - ファイル添付: 見積/契約/関連資料。既存のアップロード機能と同一UI/命名規則。
  - 自動入力補助: チャットで文脈入力 → 要約してフォーム初期値を生成。
- 承認ルート決定
  - 金額・部門・カテゴリに応じたルートマッピング（設定ファイル化）。例: 100万円超は部長→本部長→経理。
  - 代理承認設定（期間/承認限度額/対象部門）。
  - 例外ルート: 緊急稟議（ショートカットルート、後追い承認必須）。
- 承認操作
  - ステータス: Draft / 申請中 / 承認待ち(段階別) / 差戻し / 却下 / 承認済 / 失効。
  - 差戻し/却下時は理由コメント必須。再申請時は差分メモ欄。
  - 期限リマインド: 24h/72h で未処理承認者へ通知。期限超過でエスカレーション（上位承認者へ）。
  - 承認画面から添付プレビュー/ダウンロード。改ざん防止のため承認後の添付差し替えは禁止（新バージョンとして再申請）。
- 通知・チャット連携
  - 承認依頼/リマインド/差戻し/完了をメール・Slack DM で送信。チャットカードからワンクリック承認/差戻し。
  - システム内通知センターで未処理件数を表示。
- 検索・参照
  - 稟議ID、件名、申請者、部門、金額レンジ、期間、ステータスでフィルタ検索。
  - 承認履歴タイムライン（誰がいつ何をしたか）。
  - 紐づくファイルメタデータ（DocType/相手先/期間）で横断検索。
- レポート/ダッシュボード
  - 承認リードタイム、差戻し率、カテゴリ別件数、承認遅延ランキング。
  - 月次/四半期のCSV エクスポート（監査用）。IP/ユーザ/操作種別付き。
- 権限・セキュリティ
  - 起票者は自分の稟議のみ編集/閲覧。承認者は担当案件のみ操作。管理者は全件閲覧・設定変更。
  - 添付は承認済み後に改ざん不可（ハッシュ保存）。アクセスはロールベース＋テナント分離。
  - 監査ログ（操作/フィールド変更/添付閲覧）を改ざん防止領域へ書き込み。
- 例外対応
  - オフライン承認（メール承認リンクが使えない場合の代替: 管理画面で代理登録）。
  - 緊急時の暫定承認（後追い正式承認を必須化し、期限を設定）。

## 7. 非機能要件
- 可用性: 平日営業時間帯 99.9%。承認操作は 3 秒以内、一覧検索は 5 秒以内目標。
- 信頼性: 障害時のリトライ/遅延通知。二重送信防止の冪等キー。
- セキュリティ: OIDC/SAML SSO、TLS1.2+、機密度タグに応じた閲覧制限。IP 制限（社内/ゼロトラスト）。
- 監査: 7 年保管（監査/法令要件に合わせ設定可能）。エクスポート可能な監査ログ。
- プライバシー: 個人情報最小化。チャットに金額表示する場合はマスキングオプション。

## 8. データ項目（主なもの）
- 稟議: id, title, category, amount_tax_excl, tax, amount_tax_incl, currency, vendor, purpose, start_date, end_date, payment_terms, budget_code, project_code, requester_id, department, created_at, desired_approval_date
- 状態: status, step_order, current_approver_ids, next_approver_ids, due_at, escalated_to, rejection_reason, sendback_reason
- 添付: file_id, s3_object_key, hash, version, created_by
- ベンチマーク参照: freee のステップ/権限のマッピング表（外部連携なし、差異レビュー用）。
- 監査: actor_id, action, target_type, target_id, diff_json, ip, user_agent, occurred_at

## 9. UI/UX要件
- チャットからモーダル/ドロワーで起票フォームを開く。必須項目のバリデーションと進捗ガード。
- 承認カード（チャット/通知）に主要項目（件名/金額/相手先/期日）と「承認/差戻し/コメント」を配置。
- ステータスバッジと進行バーで段階を可視化。リマインドまでの残時間表示。
- 添付プレビュー（PDF/画像）。複数バージョンがある場合はタブ切替で表示。
- アクセシビリティ: キーボード操作対応、色覚バリアフリー配色、モバイルでも承認操作可能。

## 10. 運用・管理要件
- 承認ルート設定を管理画面でCSVインポート/編集可能。テスト送信機能でルート確認。
- 停止/メンテナンス時の告知と申請停止モード。停止中に送信されたリクエストはキューへ。
- monthly/quarterly レポートの自動生成と配信（経理/内部統制宛）。
- バックアップ/DR: 稟議データ・ログ・添付は日次バックアップ、RPO<=24h, RTO<=4h 目標。

## 11. 移行要件
- 既存稟議データの取り込みが必要な場合は範囲・形式を定義（ベンチマークとして freee の設計を参照）。ID マッピングを保持し、リンク切れを防止。移行後30日間は整合性を検証。
- エンドユーザ向けの利用開始手順と FAQ を準備。承認者向けショートマニュアル（2–3ページ）を配布。

## 12. リスクと対応
- ベンチマークとの差異: freee フローとの差異を定期レビューし、必要な改善を計画。
- 承認遅延: SLA/リードタイムをダッシュボード化し、エスカレーションルールを定義。
- 権限不整合: 週次でロール設定の差分チェックを自動実行。
- 監査要件未達: 操作ログの必須項目・保管期間・改ざん防止方式を監査担当と確定する。

## 13. 未決事項（要確認）
- freee ベンチマークから採用する承認ステップ/ガバナンス項目の範囲。
- 金額閾値と部門別ルートの具体値、緊急稟議の承認期限・適用条件。
- 代理承認のルール（期間、対象範囲、上限額）。
- 添付の暗号化・外部共有ポリシーの適用有無。
- 監査ログの保管先（社内SIEM/外部ストレージ）とフォーマット。
